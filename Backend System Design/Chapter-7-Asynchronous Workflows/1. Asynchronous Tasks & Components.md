# Asynchronous Tasks & Components
- **Message brokers**
	- Routes and manages messages and message queues.
	- Eg: Kafka, RabbitMQ
- **Message Queues**
	- Holds messages (jobs) until a worker is ready to process them.
	- Eg: RabbitMQ, Amazon SQS
- **Workers**
	- Pull jobs from and performs the work
	- Eg: Kubernetes (worker management)

## Questions:
  1. **In an asynchronous workflow architecture, what are the main components and how do they interact?**
	  1. An asynchronous workflow consists of: 
		  1.  A service/server that creates jobs for long-running tasks, 2.
		  2. A message broker that routes and manages messages and queues, 
		  3. Message queues that hold jobs in a line until workers are available, and 
		  4. A worker pool that pulls jobs from the queue and executes them. The flow is: `service creates a job → message broker receives it → job is placed in a queue → available worker pulls and processes the job → worker notifies when complete`.
2. **What is back pressure in the context of asynchronous workflows?**
	1. Back pressure is a concept where the system slows down the rate of incoming jobs when queues are filling up faster than workers can process them. Instead of letting queues grow indefinitely (which could crash the system), back pressure allows the service to reject new jobs or send errors to clients until the queues clear up, preventing system-wide failures.
3. **What types of tasks are good candidates for asynchronous workflows?**
	1. Computationally expensive tasks that would otherwise block system responsiveness are good candidates, including: video uploads and processing, image processing, generating reports or PDFs, payment processing, music processing, preparing zip files for download, LLM queries, and image generation. These tasks take significant time and would decrease system availability if processed synchronously.
4. **What is the most common mistake when implementing asynchronous workflows?**
	1. The most common mistake is overusing async workflows by putting everything into queues. While it's tempting because it improves surface-level performance and makes requests return quickly, it adds significant complexity and maintenance overhead. Async workflows should only be used when necessary, as they require additional infrastructure (message brokers, queues, worker pools) and make debugging more difficult when things go wrong.
5. **What are the key advantages and disadvantages of using asynchronous workflows for system scaling?**
	1. **Advantages**: Makes scaling very easy - you can add more queues, scale worker pools up or down, create dedicated worker pools for specific job types, and dynamically adjust resources based on queue depth and job types. You can pinpoint slow parts of the system and solve them quickly.
	2. **Disadvantages**: Adds significant complexity to the system, making it harder to reason about and debug. Issues like lost jobs, crashed workers, or queue failures become more difficult to track down. Maintenance overhead increases substantially.
